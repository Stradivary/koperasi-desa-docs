# 16. Infrastructure Stack

This section describes the deployment pattern and runtime expectations for the app.

## Platform architecture

- **Frontend**: static or SPA assets generated by TanStack Start, served from a CDN or edge host.
- **Terminal runtime**: browser-based terminal logic executed locally in the user’s device. The core transaction flow and card validation remain local-first.
- **Backend**: optional lightweight sync/orchestration endpoints for session grant bootstrap, remote reconciliation, audit ingestion, and tenant metadata. The system must not require a full server-side transaction engine to operate.
- **Data stores**: primary persistence is local to the terminal, while any remote storage is optional and used only for distribution, sync, and audit.

## Deployment pattern

- Serve frontend assets from **Cloudflare Pages** as the default production host.
- Treat Cloudflare primarily as a static distribution platform and CDN for the terminal web app.
- If remote APIs are used, run them on **Cloudflare Pages Functions / Workers runtime** or equivalent edge-friendly runtimes, but only as an optional sync/orchestration layer.
- Use **IndexedDB** as the browser-local persistent store for offline terminal state, tenant replicas, and reconciliation outbox.
- Use **Drizzle** as a typed local persistence layer atop IndexedDB in browser environments.
- Use **SQLite** for local-first native or desktop alternatives where a durable file-backed local database is preferable.
- Remote server-side persistence (e.g. Cloudflare D1 / KV) is optional and should not become the primary store for active terminal transactions.
- Alternative static hosts or local-first deployment models are allowed and encouraged when they preserve offline-first operation and simple tenant tracking.

## Local-first storage and multitenancy

- Store per-tenant local replicas in IndexedDB, including card cache, policy cache, operator session snapshot, and unsent reconciliation events.
- Use a simple `tenantId` tracking model to scope local data, session grants, and UI state.
- Require explicit tenant selection after login for multi-tenant accounts.
- Keep the terminal capable of performing reads and writes within a valid local grant window even without remote connectivity.

## Latency requirements

| Operation | Target latency | Notes |
|-----------|---------------|-------|
| Static asset delivery | < 200 ms | Optimized by CDN / Cloudflare Pages |
| Local grant validation | < 50 ms | Local-first and browser-executed |
| Reconciliation upload | < 1 s | Batch; not on the critical path |
| Tenant metadata sync | < 500 ms | Optional remote refresh |

## Availability requirements

- Offline operation must continue for terminal transactions while a valid cached session grant exists.
- The app must clearly indicate connectivity state and tenant sync status.
- Local state must remain authoritative for in-flight terminal transactions until reconciliation completes.
- Remote backend unavailability should not block core terminal functionality.

## Operational expectations

- Keep the core transaction system on the client side, with backend services only supporting optional sync, reconciliation, and tenant bootstrapping.
- See ADR-008 and ADR-012 for the local-first terminal model and Cloudflare distribution strategy.
- Use versioned deployments and explicit migration paths for card layout and API schema changes.
- If remote storage is used, it should backfill audit/reconciliation data rather than replace local terminal state.

> Note: future enhancement may introduce richer server-side reconciliation metadata, device/session enrollment support, and expanded local persistence models.
